# UI/UX & Desktop Usability Review: Deadliner

## 1. Review Scope
- **Документы:** `docs/ui_spec.md`, продуктовые требования.
- **Платформы:** Windows 10, CachyOS, Arch Linux (Wayland/X11).
- **Стек:** C++ / Qt 6 Widgets.
- **Фокус:** Нативные Desktop-паттерны, переполнения локализаций (RU/EN), реализуемость, когнитивная нагрузка.

## 2. Product Fit
Интерфейс в целом решает задачу: он позволяет отвлекать пользователя от ПК. Разделение "Перерывов" (цикличных) и "Событий" (одноразовых) логично.
**Однако есть проблема:** Конфликт между "Строгостью" и "Рабочим контекстом". Идея заблокировать кнопку "Skip" на 10 секунд в строгом режиме (Strict Mode) — это катастрофа для пользователя, который ведет демонстрацию экрана в Zoom/Teams или правит критический баг на проде. Продукт рискует стать токсичным и быть удаленным в первый же день.

## 3. UX Strengths
1. **Нативная парадигма:** Использование Sidebar (QListWidget) + QStackedWidget — это классический, пуленепробиваемый паттерн для десктопа (в стиле настроек Telegram Desktop или Windows Settings).
2. **Градация напоминаний (Escalation):** Мягкий -> Настойчивый -> Строгий. Это очень зрелый продуктовый подход.
3. **Осознанность локализации:** В спеке изначально заложен отказ от жестких ширин (`fixed width`) и учтена длина русских строк, что сэкономит десятки часов дебага.

## 4. Critical Issues

### [Critical] Токсичный Strict Mode без учета контекста
- **Что не так:** Блокировка кнопки "Skip/Пропустить" на 10 секунд в полноэкранном оверлее.
- **Почему это проблема:** Если у пользователя звонок, презентация экрана или гейминг, перехват фокуса и блокировка экрана вызовут панику и ярость.
- **Где проявится:** При любом рабочем форс-мажоре.
- **Как исправить:** Добавить авто-детекцию полноэкранных приложений / звонков (если возможно на С++ для целевых ОС). Добавить "Emergency Bypass" (например, двойной клик по невидимой зоне или горячая клавиша `Ctrl+Shift+Esc`), либо кнопку "I'm presenting", которая мгновенно сворачивает экран и ставит паузу на 1 час. 
- **Стоимость:** Medium. Реализация глобальных хуков сложна, но UI-кнопка "Snooze (In a call)" делается за 5 минут.

### [Critical] Выход ПК из сна (Sleep/Resume State Chaos)
- **Что не так:** В спеке не описано поведение при пробуждении компьютера.
- **Почему это проблема:** Если пользователь ушел на 3 часа, ПК уснул, а после пробуждения Qt-таймеры "догонят" время, на пользователя одновременно вывалятся 3 пропущенных перерыва и 2 события.
- **Где проявится:** У 100% пользователей ноутбуков.
- **Как исправить:** Отслеживать системные события `QSystemTrayIcon` или native API (PowerManagement). Если ПК спал, пересчитывать циклы перерывов от момента пробуждения (сбрасывать накопленную "усталость").
- **Стоимость:** High (в кроссплатформенном C++ API работы с питанием).

### [Critical] Невидимость трея в Linux (GNOME / Wayland)
- **Что не так:** Ставка на `QSystemTrayIcon` как на основной инструмент управления.
- **Почему это проблема:** В Arch/CachyOS дефолтный GNOME **не имеет** системного трея. Без установленного расширения AppIndicator приложение, свернутое в трей, станет недоступным.
- **Где проявится:** Arch Linux, CachyOS (GNOME Wayland).
- **Как исправить:** Никогда не убирать приложение из Alt+Tab / Taskbar в Linux, если невозможно гарантированно подтвердить наличие трея. В настройках: опция "Закрывать в трей" должна быть *отключена* по умолчанию на Linux, пока не будет проверено API.

## 5. High Priority Issues

### [High] Live Language Switch и динамические данные
- **Что не так:** Заявлено изменение языка через `ui->retranslateUi(this)`.
- **Почему это проблема:** `retranslateUi` переводит только статические строки из `.ui` форм. Все тексты вида `"Next break in: " + time` или списки сгенерированных элементов **не переведутся** автоматически. В UI будет каша из RU и EN текстов до перезапуска.
- **Где проявится:** На Dashboard и в списках.
- **Как исправить:** При перехвате `QEvent::LanguageChange` необходимо не только вызывать `retranslateUi()`, но и иметь кастомный метод `updateDynamicTexts()`, который заново форматирует все таймеры и данные.
- **Стоимость:** Medium, но требует жесткой архитектурной дисциплины.

### [High] Перегруженный Dashboard
- **Что не так:** На главном экране предложено разместить кнопки `[Edit] [Toggle]` для правил перерывов.
- **Почему это проблема:** Dashboard — это сводка (View/Status), а не панель управления (Settings). Смешивание управления и статуса усложняет восприятие.
- **Как исправить:** Убрать чекбоксы и Edit-кнопки с Dashboard. Оставить только список: "Активно: 20-20-20. Следующий через 15 мин". Все редактирование — только во вкладке Breaks.

## 6. Medium Priority Issues

### [Medium] Столкновение событий (Collisions)
- **Что не так:** Нет логики при совпадении таймера перерыва и жесткого события (Event).
- **Как исправить:** Event имеет приоритет. Если сработало Event-напоминание, таймер перерыва должен тихо сброситься (так как пользователь и так отвлекся на событие).

### [Medium] Логика кнопки `Esc`
- **Что не так:** `Esc` закрывает диалоги. Но в контексте напоминания: это "Сбросить/Пропустить" (Skip) или "Отложить" (Snooze)?
- **Как исправить:** Явно закрепить, что `Esc` работает как мягкий Snooze на минимальное дефолтное время (например, 5 минут), чтобы случайное закрытие диалога не привело к потере цикла. В Strict режиме `Esc` должен игнорироваться.

## 7. Low Priority Issues
- Вкладка **Statistics (Статистика)** избыточна для MVP. Это "хотелка", которая отнимает время от шлифовки логики таймеров. Рекомендую скрыть или вырезать из первого релиза.

## 8. Accessibility Review
- **Tab Order:** Должен быть замкнут (Focus Trap) в модальных окнах (Persistent / Strict), чтобы Tab не уходил в фон.
- **Цвет:** У кнопок "Пропустить" и "Отложить" должна быть разница в визуальном весе (Primary vs Secondary), а не только в цвете (Красный/Серый), чтобы дальтоники легко различали Primary-действие (Отложить).

## 9. Localization Review (RU/EN)
В спеке заявлены правильные принципы, но требуются уточнения по wording-парадигмам.

**Опасные пары (Требуют Pseudo-localization тестирования):**
- *Snooze* (6) -> *Отложить* (8) — Хорошо. Но *Snooze for 10 m* (14) -> *Отложить на 10 м* (16).
- *Skip* (4) -> *Пропустить* (10) — Нормально для кнопок с паддингами.
- *Quiet hours* (11) -> *Режим тишины* (12) или *Тихий час* (9) — Нормально.
- **Риск:** Составные строки. `QString("Next break in: %1").arg(time)` -> на русском: `"Следующий перерыв через: %1"`. Слишком длинно.
  - *Решение:* Сделать UI в два этажа (QLabel сверху, время снизу крупным шрифтом). Это полностью убивает проблему длины строки.
  - Пример:
    `Next break / Следующий перерыв`
    `[ 14:32 ]`

## 10. Desktop/Qt Feasibility Review
- **Layouts:** `QVBoxLayout`, `QHBoxLayout` и `QSizePolicy::Expanding` в Qt отлично справятся с задачей.
- **High DPI:** Использование SVG (`QIcon`) — абсолютно верное решение. Однако на Linux (Wayland) fractional scaling (125%, 150%) в Qt 6 может мылить шрифты в зависимости от переменных окружения (`QT_WAYLAND_DISABLE_WINDOWDECORATION`). Окно `ReminderStrictWindow` следует делать на базе чистого `QWidget`, а не `QMainWindow`, чтобы избежать багов с Wayland-рамками.

## 11. Cross-Platform Review
- **Нативные уведомления (Soft mode):**
  - Windows: `QSystemTrayIcon::showMessage` работает через Windows Toast API (отлично).
  - Linux: Работает через `libnotify`. Если демон уведомлений (dunst/mako) не запущен, они уйдут в пустоту. Обязателен фоллбэк: если `QSystemTrayIcon::supportsMessages()` возвращает `false`, Soft mode должен принудительно переключаться в Persistent Mode.

## 12. Reminder Mode Review
- Градация отличная.
- **Спорно:** В Persistent Mode (небольшое окно без рамки) нужно обеспечить, чтобы оно не крало фокус у текущего окна пользователя. В Qt это `Qt::WindowDoesNotAcceptFocus` (или `Qt::WindowStaysOnTopHint | Qt::Tool`). Иначе пользователь, печатающий текст, случайно нажмет "Пробел", и это сработает как нажатие на кнопку в диалоге напоминания.

## 13. MVP Readiness
- UI готов к MVP на **70%**.
- Требуется: убрать статистику, перенести туглы включения/выключения из Dashboard в Settings/Rules, детально расписать обработку событий Sleep/Resume.

## 14. Final Verdict
**Verdict: APPROVE WITH FIXES**

## 15. Recommended Next Changes
1. Убрать элементы управления (чекбоксы/редактирование) с Dashboard.
2. Проектировать блок с таймером в две строки (Текст сверху, время под ним) для защиты от локализационных багов.
3. Добавить флаг фоллбэка на случай отсутствия системного трея (особенно для GNOME).
4. Прописать логику `Focus Trap` и `WindowDoesNotAcceptFocus` для всплывающих напоминаний.
5. Заложить обработку события "выход из спящего режима" на уровне контроллера приложения (не UI).
